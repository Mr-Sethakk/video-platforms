## 方案对比

### 方案 A：环境变量 + .env.example（轻量级）

**原理**：敏感配置通过系统环境变量或 `.env` 文件注入，`application.yml` 只保留非敏感的开发默认值。提供 `.env.example` 模板，用户复制后填入自己的 Key。

**用户操作流程**：
```bash
# 1. 安装依赖（需自行安装 JDK17+、Maven、Node.js18+、MySQL、Redis）
# 2. 启动 MySQL 和 Redis
docker-compose up -d mysql redis    # 或手动启动

# 3. 配置环境变量
cp backend/.env.example backend/.env
# 编辑 backend/.env，填入你的 API Key

# 4. 启动后端
cd backend && mvn spring-boot:run

# 5. 启动前端
cd frontend && npm install && npm run dev

# 6. 访问 http://localhost:3000
```

**需要改造的内容**：
- [x] `application.yml` 敏感值改为 `${VAR:dev默认值}` 占位符 — **已完成**
- [x] 完善 `backend/.env.example` 模板 — **已完成**
- [ ] README 写清楚环境要求和配置步骤
- [ ] 可选：添加 `spring-dotenv` 依赖让 Spring Boot 自动读取 `.env` 文件
- [ ] 可选：前端 `frontend/.env.local.example` 模板

**✅ 优点**：
- 改动最小（已完成 80%）
- GitHub 上最常见的做法，开发者熟悉
- 灵活性高，适合有经验的开发者

**❌ 缺点**：
- 用户需要安装 5 个依赖（JDK、Maven、Node.js、MySQL、Redis）
- "我这能跑你那不行"问题 — 环境差异导致奇怪 bug
- AI 相关功能需要额外注册获取 API Key
- 对非技术人员（老师/同学）不够友好

**适合场景**：开发者协作、有技术背景的用户

---

### 方案 B：Spring Profile + 外部配置文件

**原理**：利用 Spring Boot 的多环境配置机制，敏感配置放在不被 Git 追踪的外部文件中。开发/生产环境通过 `spring.profiles.active` 切换。

**需要改造的内容**：
- [ ] 创建 `application-dev.yml`（提交到 Git）— 开发环境默认配置
- [ ] 创建 `application-prod.yml`（不提交，加入 .gitignore）— 生产环境真实配置
- [ ] 创建 `application-prod.example.yml`（提交到 Git）— 生产配置模板
- [ ] `application.yml` 中设置 `spring.profiles.active: dev`
- [ ] 通过 `--spring.config.additional-location` 加载外部敏感配置文件
- [ ] README 说明多环境切换方式

**配置结构示例**：
```
backend/src/main/resources/
├── application.yml              # 公共配置 + 激活 dev profile
├── application-dev.yml          # 开发环境（提交到 Git，含 dev 默认值）
└── application-prod.example.yml # 生产环境模板（提交到 Git）
                                 # 用户复制为 application-prod.yml 并填入 Key
```

**✅ 优点**：
- Spring Boot 原生支持，无额外依赖
- 多环境切换灵活（dev / staging / prod）
- 配置与代码分离彻底
- 适合有运维经验的人

**❌ 缺点**：
- 对新手不够友好，需要理解 Spring 配置加载机制
- 仍然需要用户安装所有环境依赖
- 改动量中等

**适合场景**：有多环境部署需求、团队有运维规范

---

### 方案 C：Docker 全容器化（⭐ 推荐）

**原理**：所有服务（前端 + 后端 + MySQL + Redis）全部 Docker 容器化。用户只需安装 Docker，一个命令启动全部服务。敏感信息通过 Docker 的 `.env` 文件统一管理。

**用户操作流程**：
```bash
# 1. 克隆项目
git clone <repo-url> && cd summer_internship

# 2. 配置环境变量（唯一需要编辑的文件）
cp .env.example .env
# 编辑 .env，填入你的 API Key（可选，不填则 AI 功能不可用但其他功能正常）

# 3. 一键启动
docker-compose up -d

# 4. 访问 http://localhost:3000
```

**需要改造的内容**：

```
新增文件清单：
├── .env.example                    # 统一的环境变量模板（提交到 Git）
├── docker-compose.yml              # 完善：加入 backend + frontend 服务
├── backend/Dockerfile              # 后端多阶段构建（maven build → JDK run）
├── frontend/Dockerfile             # 前端多阶段构建（npm build → node run）
├── frontend/nginx.conf             # 可选：生产模式用 nginx  serve 静态文件
└── .dockerignore                   # Docker 构建排除文件
```

**Docker Compose 架构**：
```
┌──────────────────────────────────────────────────┐
│                  docker-compose.yml               │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌───────────────┐  │
│  │  MySQL   │  │  Redis   │  │   Backend      │  │
│  │  :3306   │  │  :6379   │  │   :8080        │  │
│  │          │  │          │  │   Spring Boot  │  │
│  └────┬─────┘  └────┬─────┘  │   (mvn build)  │  │
│       │             │         └───────┬───────┘  │
│       └──────┬──────┘                 │          │
│              │                        │          │
│              ▼                        ▼          │
│  ┌───────────────────────────────────────────┐   │
│  │              Frontend                      │   │
│  │              :3000                         │   │
│  │              Next.js (npm run dev/prod)    │   │
│  └───────────────────────────────────────────┘   │
│                                                   │
│  全部通过 .env 文件注入环境变量                      │
└──────────────────────────────────────────────────┘
```

**✅ 优点**：
- ⭐ **体验最佳**：用户只需 Docker，无需安装 JDK/Maven/Node/MySQL/Redis
- 环境完全一致，彻底解决"我这能跑你那不行"
- 一条命令启动/停止
- AI 功能优雅降级：不配 Key 也能用，只是 AI 功能不可用
- GitHub 上现代项目的标配

**❌ 缺点**：
- 改造工作量最大（约 1-2 天）
- 用户需要安装 Docker Desktop（~500MB）
- Java Docker 镜像构建较慢（首次约 3-5 分钟）
- Docker 镜像体积较大（后端 ~400MB + 前端 ~200MB）

**适合场景**：课程作业展示、任何人快速体验、README 演示

## 🎯 我的建议

### 建议采取"渐进式"策略：

1. **立即** → 方案 A（已完成 80%，补个 README 即可作为当前版本交付）
2. **本周** → 方案 C（写 Dockerfile + 完善 docker-compose，作为项目的"一键体验"入口）
3. **可选** → 方案 B（如果后续有 staging/prod 多环境需求再上）

这样你既能快速交付，又能逐步提升项目质量。方案 A 保证现在就能用，方案 C 让项目更专业。

### 关于 API Key 的额外建议

对于 AI 功能（DeepSeek 聊天 + DashScope 海报识别），建议采用**优雅降级**策略：
- 未配置 Key → 前端显示"AI 助手暂不可用，请配置 API Key"的友好提示
- 配置了 Key → 正常使用
- 这样即使不配 Key，项目的电影浏览/搜索/收藏等核心功能完全正常
